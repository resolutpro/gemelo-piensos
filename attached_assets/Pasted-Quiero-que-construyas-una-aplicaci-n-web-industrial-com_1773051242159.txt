Quiero que construyas una aplicación web industrial completa, funcional y bien estructurada para un proyecto de digitalización de una fábrica de piensos. No quiero una landing page, sino una aplicación real tipo dashboard operativo, con frontend y backend, base de datos, API, interfaz responsive y estructura mantenible.

CONTEXTO DEL NEGOCIO
La aplicación pertenece a una fábrica de piensos para animales. La fábrica ya tiene un SCADA antiguo que controla la producción física, básculas, electroválvulas, mezcladora, molinos y tolvas. Esta nueva web NO debe conectarse ni controlar directamente el SCADA en esta fase. Debe funcionar como un sistema paralelo de apoyo a la decisión, un “shadow pilot” o “gemelo digital en paralelo”, sin riesgo operativo.

OBJETIVO DE LA WEB
La web debe servir para:
1. Registrar la calidad real de materias primas al entrar en fábrica mediante análisis NIR.
2. Registrar y visualizar sensores IoT ambientales de polvo, temperatura y humedad.
3. Simular mezclas de pienso antes de fabricar físicamente.
4. Predecir la calidad esperada del pienso final a partir de los lotes e ingredientes seleccionados.
5. Registrar la verificación real del pienso terminado con un nuevo análisis NIR.
6. Comparar predicción vs realidad y generar recomendaciones para futuras mezclas.
7. Mantener trazabilidad completa desde la llegada del camión hasta el producto terminado.
8. Lanzar alertas por riesgo ambiental o por desviaciones de calidad.
9. Mostrar dashboards claros para operarios y responsables.

REQUISITOS CLAVE
- La fábrica recibe materias primas como maíz, soja, trigo, cebada, etc.
- El problema actual es que no se conoce la calidad nutricional real en tiempo real al recibir la materia prima.
- El operario debe poder registrar un lote de entrada y asociarle un análisis NIR.
- La fábrica tiene riesgos ambientales:
  - polvo en suspensión en zonas como molinos y elevadores
  - humedad y temperatura en almacén de producto terminado
- La web debe ayudar a reducir la dependencia de un analizador NIR inline caro y abrasivo.
- La web debe aprender comparando la predicción de la mezcla con el análisis real del producto terminado.
- En esta fase, la interacción con el proceso productivo será manual: el usuario introduce lotes, ingredientes, kilos y resultados.
- NO quiero datos falsos, inventados o de ejemplo. La aplicación debe trabajar únicamente con datos guardados en la base de datos.
- Si la base de datos está vacía, la app debe mostrar estados vacíos correctos y mensajes útiles como “Todavía no hay lotes registrados”.

STACK TECNOLÓGICO
Quiero una arquitectura moderna, clara y mantenible.
Usa preferiblemente:
- Frontend: React + TypeScript
- UI: Tailwind CSS
- Backend: Node.js + TypeScript
- API: REST
- Base de datos: PostgreSQL o SQLite si es más práctico en Replit
- ORM: Prisma
- Gráficos: Recharts
- Formularios: React Hook Form + Zod

Si consideras mejor usar Next.js full stack con App Router y TypeScript para simplificar el proyecto, úsalo. Prioriza que funcione bien y que el código quede ordenado.

MÓDULOS OBLIGATORIOS

1. Dashboard principal
Debe mostrar información real procedente de la base de datos:
- número de lotes recibidos
- últimas analíticas NIR
- alertas activas
- estado ambiental por zonas
- últimas simulaciones realizadas
- desviaciones entre predicción y resultado real

Si no hay datos, mostrar tarjetas vacías con textos claros, sin inventar cifras.

2. Módulo de recepción de materias primas
Debe permitir:
- crear lote de entrada
- registrar proveedor
- fecha y hora
- tipo de materia prima
- código de lote
- cantidad
- ubicación o tolva destino
- observaciones
- registrar análisis NIR asociado

Campos NIR sugeridos:
- humedad
- proteína
- grasa
- almidón
- fibra
- cenizas
- otros parámetros configurables

Debe existir una vista de listado y una ficha detallada del lote.

3. Módulo de sensores IoT y monitorización ambiental
Debe permitir visualizar:
- sensores de polvo por zona
- sensores de temperatura y humedad por zona
- estado del sensor
- último dato recibido
- histórico por rango de fechas
- nivel de alerta

Zonas sugeridas:
- recepción
- piqueras
- molinos
- elevadores
- mezcladora
- almacén de producto terminado
- ensacado

Para esta primera versión, implementa ingestión de datos de sensores de estas dos formas:
A. entrada manual desde la interfaz
B. endpoint API para recibir datos reales desde dispositivos o integraciones futuras

Cada sensor debe tener:
- nombre
- tipo
- zona
- unidad de medida
- estado
- batería opcional
- fecha del último dato
- umbrales de alerta

4. Módulo de simulación de mezcla / gemelo digital
Debe permitir al usuario:
- seleccionar una receta teórica o crear una mezcla manual
- añadir ingredientes/lotes
- indicar kilos por ingrediente
- ver automáticamente la composición nutricional estimada de la mezcla final

La simulación debe calcular una predicción a partir de la composición de los lotes seleccionados.
Por ejemplo:
si se mezclan 3000 kg de maíz con 2000 kg de soja, la app debe calcular proteína estimada, humedad estimada, etc. como media ponderada según kilos y composición del lote.

Debe mostrar:
- tabla de ingredientes
- kilos por ingrediente
- porcentaje sobre total
- composición esperada
- resultado nutricional predicho
- advertencias si no se cumplen objetivos

Debe permitir fijar objetivos de receta, por ejemplo:
- proteína mínima
- humedad máxima
- grasa objetivo

La interfaz debe indicar visualmente:
- cumple
- cerca del límite
- no cumple

5. Módulo de producción física registrada
Como la web no controla el SCADA, debe existir un paso donde el usuario indique que la mezcla ha sido fabricada físicamente.
Debe poder:
- marcar una simulación como “fabricada”
- registrar fecha/hora real
- registrar observaciones
- registrar lote de producto terminado generado

6. Módulo de verificación del producto final
Tras fabricar, el usuario debe poder registrar una nueva analítica NIR del pienso terminado.
La aplicación debe:
- asociar la analítica al lote de producto terminado
- comparar lo predicho con lo medido realmente
- mostrar desviaciones absolutas y porcentuales
- guardar histórico de comparaciones

Debe mostrarse algo como:
- proteína predicha
- proteína real
- desviación

7. Módulo de recomendaciones
A partir de la comparación entre predicción y realidad, la app debe generar recomendaciones simples y útiles.
No hace falta IA compleja al principio.
Implementa primero una lógica basada en reglas y deja preparada la estructura para evolucionar más adelante.

Ejemplos:
- “La proteína real quedó por debajo de la predicción”
- “Revisar calidad del lote de soja”
- “Sugerencia para próxima mezcla similar: ajustar porcentaje de soja”
- “El lote de maíz presenta humedad alta”

Haz que el motor de recomendaciones sea modular.

8. Módulo de trazabilidad
Debe permitir recorrer la historia completa de un lote:
- entrada de materia prima
- proveedor
- analítica NIR de entrada
- uso en simulaciones
- mezcla fabricada
- lote de producto final
- analítica final
- sensores ambientales relacionados por fechas y zona
- incidencias o alertas asociadas

Debe haber una vista cronológica clara y también una vista de detalle por lote.

9. Módulo de alertas
Debe generar alertas por:
- polvo por encima de umbral
- humedad relativa por encima de umbral
- temperatura fuera de rango
- sensor sin datos
- desviación grande entre predicción y realidad
- lote fuera de especificación

Crea:
- listado de alertas
- alertas activas y resueltas
- severidad: info / warning / crítica
- filtros por zona, fecha y tipo
- opción de marcar resuelta

10. Módulo de administración básica
Sin roles, pero sí configuración básica:
- gestionar materias primas
- gestionar proveedores
- gestionar zonas
- gestionar sensores
- configurar umbrales de alerta
- configurar parámetros nutricionales visibles
